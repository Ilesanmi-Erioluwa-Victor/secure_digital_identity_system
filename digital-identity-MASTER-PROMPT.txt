PROJECT TITLE
-------------
Design and Implementation of a Secure Digital Identity System
(A Case Study of Delta State Polytechnic Library)

ABSTRACT
--------
The management of user identities in institutional settings such as
a polytechnic library presents significant challenges including
unauthorized access, identity fraud, and inefficient manual
verification processes. This project presents the design and
implementation of a Secure Digital Identity System for the Delta
State Polytechnic Library. The system provides each library user
(student, staff, or external member) with a unique verifiable
digital identity in the form of a QR-code-backed digital ID card.
It enables secure login using multi-factor authentication, manages
role-based access to library resources, tracks identity verification
activities, and generates an audit trail of all access events.
Administrators can issue, revoke, suspend, and renew digital
identities. The system replaces physical ID card dependency with
a secure, centralized, and easily verifiable digital identity
infrastructure that reflects real-world institutional identity
management practices.

WHY THIS STACK?
---------------
- JWT (Access + Refresh Tokens): Access tokens expire in 15 minutes
  for security. Refresh tokens (7 days) silently renew sessions
  without re-login, balancing security and usability — a standard
  practice in modern identity systems.

- otplib + speakeasy: Implements Time-based One-Time Passwords
  (TOTP) for two-factor authentication, the same standard used by
  Google Authenticator. This adds a second verification layer
  beyond passwords, critical for an identity system.

- qrcode (server-side): QR codes are generated server-side and
  embedded in digital ID cards. Each QR encodes a signed, encrypted
  verification URL. When scanned by a librarian, it hits the backend
  verification endpoint which confirms the identity's validity,
  status, and ownership — without exposing raw user data in the QR.

- MongoDB: Flexible schema suits identity records which can vary
  across user types (student has matric number, staff has staff ID,
  external has NIN). Document model stores identity metadata,
  verification history, and access logs naturally.

- crypto (built-in): Used for generating cryptographically secure
  random tokens for email verification, password reset, and identity
  verification links — no third-party dependency needed.

WHAT EACH FEATURE DOES
-----------------------

1. DIGITAL IDENTITY ISSUANCE
   The admin registers a user and issues them a unique Digital
   Identity. Each identity has: a system-generated Digital ID Number
   (DID-POLY-XXXXXX), a profile photo, biographic details, role
   and access level, issue date, expiry date, and a QR code. The QR
   code encodes a signed verification URL unique to that identity.
   The digital ID is downloadable as a PDF card resembling a
   physical library ID card.

2. MULTI-FACTOR AUTHENTICATION (MFA)
   Login requires two factors:
   Factor 1 — Email + Password (bcrypt-verified)
   Factor 2 — OTP sent to the user's registered email (6-digit,
   expires in 10 minutes) OR TOTP from an authenticator app if
   the user has enabled app-based 2FA.
   MFA can be toggled per user by admin. High-privilege roles
   (Admin, Librarian) always require MFA — it cannot be disabled
   for them.

3. QR CODE IDENTITY VERIFICATION
   Librarians use a built-in QR scanner page to scan a member's
   digital ID QR code. The scan decodes the verification URL and
   hits the backend /api/identity/verify/:token endpoint. The
   backend checks: identity exists, status is Active, not expired,
   and the verification token matches the stored signed hash. The
   response shows: photo, name, ID number, role, access level,
   expiry date, and a large VALID / INVALID / SUSPENDED / EXPIRED
   status banner. Every scan is logged in the AccessLog.

4. IDENTITY LIFECYCLE MANAGEMENT
   Admins manage the full identity lifecycle:
   - Issue: create new digital identity for a user
   - Activate: set status to Active (default after issue)
   - Suspend: temporarily block access (with reason)
   - Revoke: permanently invalidate identity
   - Renew: extend expiry date (generates new QR token)
   - Replace: issue new identity (e.g. after name change)
   Each lifecycle event is logged with the admin's ID, reason,
   and timestamp.

5. ROLE-BASED ACCESS LEVELS
   Each identity carries an Access Level that determines which
   library resources the user can access:
   - Level 1 (Student): borrow books, access reading room
   - Level 2 (Staff): Level 1 + access reference section,
     request inter-library loans
   - Level 3 (Librarian): Level 2 + manage collections
   - Level 4 (Admin): full system access
   The access level is encoded in the digital identity and
   displayed during QR verification.

6. ACCESS LOG / AUDIT TRAIL
   Every identity verification scan, login attempt (success or
   failure), identity status change, and admin action is recorded
   in the AccessLog with: user/identity ID, action type, outcome
   (success/failure/suspicious), IP address, device/browser info,
   and timestamp. Admins can view, search, filter, and export
   the full audit trail.

7. IDENTITY EXPIRY AND RENEWAL AUTOMATION
   Each digital identity has an expiry date (default 1 year from
   issue). A node-cron job runs daily and:
   - Sends expiry warning email 30 days before expiry
   - Sends final warning 7 days before expiry
   - Automatically sets status to Expired on the expiry date
   Admin can renew expired identities — generates a new expiry
   date and a new QR verification token.

8. DASHBOARD AND ANALYTICS
   Admin dashboard shows: total identities issued, active count,
   suspended count, expired count, today's verifications, recent
   access log entries, identity status breakdown chart, and
   monthly issuance trend chart.

9. USER SELF-SERVICE PORTAL
   Authenticated users can: view their own digital identity card,
   download their PDF ID card, update their profile photo, change
   their password, enable/disable app-based 2FA (with QR setup
   for Google Authenticator), and view their own access/scan history.

10. SECURITY FEATURES
    - Bcrypt password hashing (saltRounds: 10)
    - Short-lived JWT access tokens (15 minutes)
    - Refresh token rotation (7 days, stored in httpOnly cookie)
    - Rate limiting on login endpoint (max 5 attempts per 15 min)
    - Account lockout after 5 failed login attempts (30 min lock)
    - All QR verification tokens are HMAC-signed with a server secret
    - Identity verification tokens are hashed before storage
    - Full audit trail of all actions
    - CORS restricted to frontend origin

PROJECT OVERVIEW FOR CODEX
---------------------------
Build a full-stack Secure Digital Identity System for a polytechnic
library. The system issues verifiable digital identity cards with
QR codes to library users. It supports multi-factor authentication
(email OTP + optional TOTP), identity lifecycle management (issue,
activate, suspend, revoke, renew), QR-based identity verification
at library entry points, role-based access levels, a full audit
trail of all access events, and automated expiry management.

Tech Stack: React.js (Vite) + Tailwind CSS + Node.js + Express.js
+ MongoDB (Mongoose) + JWT (access + refresh) + bcryptjs +
otplib/speakeasy + qrcode + Multer + Nodemailer + pdfkit +
node-cron + Recharts

BACKEND API ROUTES
================================================================

AUTH — /api/auth
  POST  /login                  Step 1: email + password → send OTP
  POST  /verify-otp             Step 2: submit OTP → return tokens
  POST  /verify-totp            Step 2 alt: submit TOTP code → tokens
  POST  /refresh                Use refresh token → new access token
  POST  /logout                 Revoke refresh token
  POST  /forgot-password        Send reset email
  POST  /reset-password/:token  Reset password
  GET   /me                     Get current user (from access token)

IDENTITY — /api/identity
  POST  /                       Issue new identity (Admin only)
  GET   /                       All identities paginated (Admin, Librarian)
  GET   /my                     Current user's own identity
  GET   /:id                    Single identity full details
  PUT   /:id                    Update identity details (Admin)
  POST  /:id/suspend            Suspend with reason (Admin)
  POST  /:id/activate           Re-activate suspended identity (Admin)
  POST  /:id/revoke             Revoke permanently (Admin)
  POST  /:id/renew              Renew + regenerate QR token (Admin)
  GET   /:id/download           Download PDF ID card
  POST  /:id/photo              Upload/update profile photo (Multer)

VERIFY — /api/verify
  GET   /:token                 Verify QR token (public — no auth needed)
                                Returns identity status + display info
                                Logs every scan to AccessLog

USERS — /api/users (Admin only)
  POST  /                       Create new user account
  GET   /                       All users paginated
  GET   /:id                    Single user
  PUT   /:id                    Update user
  POST  /:id/unlock             Unlock locked account
  DELETE /:id                   Deactivate user

TOTP — /api/totp (authenticated users)
  GET   /setup                  Generate TOTP secret + QR for setup
  POST  /enable                 Confirm TOTP code to enable app 2FA
  POST  /disable                Disable TOTP (requires password confirm)

ACCESS LOGS — /api/logs
  GET   /                       All logs — Admin + Librarian (paginated)
  GET   /my                     Current user's own logs
  GET   /export                 Export logs as PDF (Admin only)

REPORTS — /api/reports (Admin only)
  GET   /summary                Total identities by status
  GET   /monthly-issuance       Monthly identity issuance trend
  GET   /access-summary         Login/scan activity last 30 days
  GET   /role-breakdown         Identity count by role
  GET   /export/identities      Export identity register PDF
  GET   /export/access-logs     Export access log report PDF

MFA AUTHENTICATION FLOW (detailed)
================================================================

STEP 1 — /api/auth/login:
  1. Find user by email
  2. Check account lock (lockUntil > now → return 423 with unlock time)
  3. Compare bcrypt password
  4. If wrong: increment loginAttempts, if >= 5 → set lockUntil,
     log ACCOUNT_LOCKED to AccessLog, return 401
  5. If correct: reset loginAttempts to 0
  6. If isTOTPEnabled: return { mfaRequired: true, method: 'totp', email }
  7. If isMFAEnabled (email OTP):
     a. Generate 6-digit OTP using generateOTP.js
     b. Hash OTP and store on user (otpCode, otpExpiry = now + 10min)
     c. Send OTP email
     d. Log OTP_SENT to AccessLog
     e. Return { mfaRequired: true, method: 'email', email }
  8. Log LOGIN_SUCCESS (rare — only if admin disabled MFA entirely)

STEP 2a — /api/auth/verify-otp:
  1. Find user by email
  2. Check otpExpiry not passed → 400 "OTP expired"
  3. Compare submitted OTP hash to stored hash
  4. If wrong: log OTP_FAILED, return 401 "Invalid OTP"
  5. If correct: clear otpCode/otpExpiry, log OTP_VERIFIED
  6. Generate access token (15min) + refresh token (7days)
  7. Store hashed refresh token in RefreshToken collection
  8. Set refresh token as httpOnly cookie
  9. Return access token in response body

STEP 2b — /api/auth/verify-totp:
  1. Find user by email, get totpSecret (decrypt if encrypted)
  2. Use speakeasy.totp.verify(secret, token, window: 1)
  3. If invalid: return 401
  4. If valid: same token generation flow as STEP 2a

TOKEN REFRESH — /api/auth/refresh:
  1. Read refresh token from httpOnly cookie
  2. Hash it, find in RefreshToken collection
  3. Validate not revoked, not expired
  4. Generate new access token
  5. Rotate refresh token (delete old, create new)
  6. Return new access token

Axios interceptor on client:
  On 401 response: call /api/auth/refresh silently
  If refresh succeeds: retry original request with new token
  If refresh fails: clear auth state, redirect to /login

QR VERIFICATION TOKEN SYSTEM
================================================================

When identity is issued or renewed:
  1. Generate raw verify token: 32 random bytes → hex string
  2. Create HMAC signature: crypto.createHmac('sha256', IDENTITY_VERIFY_SECRET)
     .update(rawToken + identity._id).digest('hex')
  3. verifyTokenRaw = rawToken (embedded in QR URL — never stored in DB)
  4. Store verifyToken = SHA-256 hash of (rawToken + HMAC) in Identity document
  5. QR code URL format:
     CLIENT_URL/verify/[rawToken].[hmacSignature]
  6. Generate QR code image from URL using qrcode npm package
  7. Store base64 QR image in Identity.qrCodeImage

When QR is scanned:
  GET /api/verify/:token
  where :token = [rawToken].[hmacSignature]
  1. Split token on '.'
  2. Recompute HMAC from rawToken
  3. Compare to submitted hmacSignature (constant-time compare)
  4. If mismatch: log QR_SCAN_INVALID, return INVALID
  5. Recompute stored hash, find Identity where verifyToken matches
  6. If not found: log QR_SCAN_INVALID, return INVALID
  7. Check Identity.status:
     Active + not expired → log QR_SCAN_VALID → return VALID + details
     Expired             → log QR_SCAN_EXPIRED → return EXPIRED + details
     Suspended           → log QR_SCAN_SUSPENDED → return SUSPENDED + details
     Revoked             → log QR_SCAN_INVALID → return REVOKED
  8. Return: { status, identity: { photo, fullName, digitalIDNumber,
     role, accessLevel, department, issueDate, expiryDate } }

NOTE: The verification endpoint is PUBLIC (no auth required) so that
a QR scanner device or even a phone browser can verify without login.

BUSINESS RULES
================================================================

1.  Admin and Librarian roles always require MFA — cannot disable
2.  Each user can have only ONE active identity at a time
3.  Revoking is permanent — no re-activation after revocation
4.  Renewing creates a new verifyToken + QR code + resets expiry
5.  Suspended identities block QR verification but user can still login
6.  Expired identities: user can login but QR verification fails
7.  Profile photo is REQUIRED before identity can be issued
8.  Identity photo change requires admin action (PUT /api/identity/:id)
9.  Soft-delete users only — deactivated users cannot login
10. If user is deactivated, their identity auto-suspends
11. All OTPs expire in 10 minutes, max 3 resend requests per hour
12. Access tokens expire in 15 minutes (security standard)
13. Refresh tokens rotate on every use (prevents token theft)
14. Failed login attempts and lockouts are always logged
15. QR verification logs every scan regardless of outcome
16. Export of access logs requires Admin role

DEFENSE NOTES
================================================================

Q: What is a digital identity system and why is it needed for a library?
A: A digital identity system provides each user with a unique, verifiable
electronic credential that proves who they are. In a library setting,
it replaces physical ID cards which can be forged, lost, or borrowed by
others. The digital system ties identity to a cryptographically signed
QR code that can only be validated against the live database, making
impersonation virtually impossible.

Q: How does multi-factor authentication work in your system?
A: MFA requires two independent proofs of identity. Factor one is the
password, which proves knowledge. Factor two is either a time-limited
OTP sent to the user's registered email, or a TOTP from an authenticator
app, which proves possession of the registered device. Even if a password
is stolen, an attacker cannot login without also controlling the user's
email or phone.

Q: How does the QR code verification work — how do you know it hasn't been forged?
A: The QR code encodes a URL containing two parts: a random token and an
HMAC signature generated using a secret key stored only on the server.
When the QR is scanned, the server recomputes the HMAC from the submitted
token and compares it to the submitted signature using constant-time
comparison to prevent timing attacks. If they don't match, the QR is
flagged as tampered. If they match, the system looks up the identity by
a stored hash of the token combination. This means the QR cannot be
fabricated without knowing the server's secret key.

Q: Why do you use short-lived access tokens (15 minutes)?
A: In identity systems, shorter token lifetimes minimize the window of
exposure if a token is intercepted. A stolen access token becomes useless
after 15 minutes. Refresh tokens, which last 7 days but are stored in
httpOnly cookies and rotated on every use, handle seamless session renewal
without requiring re-login.

Q: What is refresh token rotation and why is it important?
A: Every time a refresh token is used to get a new access token, the old
refresh token is deleted and a new one is issued. This means a stolen
refresh token becomes invalid after the legitimate user makes any request.
It provides forward secrecy — past tokens cannot be reused even if
intercepted — which is a critical security property for an identity system.

Q: How do you prevent brute-force attacks on the login page?
A: Two mechanisms work together. First, express-rate-limit restricts the
/login endpoint to a configurable maximum of attempts (default 5) per IP
address within a 15-minute window, returning 429 Too Many Requests after
the limit is reached. Second, at the user level, after 5 consecutive failed
login attempts, the account is locked for 30 minutes (configurable in
Settings). Both events are logged in the AccessLog for admin review.

Q: What is the identity lifecycle and why does it matter?
A: An identity lifecycle tracks every state change a digital identity goes
through: Issued → Active → Suspended/Revoked/Expired → Renewed. Each
transition is logged with who performed it, why, and when — stored in the
identity's lifecycleHistory array. This is essential for accountability
in an institutional setting, allowing administrators to audit why an
identity was suspended and by whom, which reflects real-world identity
governance practices.

Q: Why is there a distinction between Suspended and Revoked?
A: Suspension is temporary — it blocks QR verification and library access
but can be reversed by an admin once the issue is resolved (e.g. after a
student clears outstanding dues). Revocation is permanent and irreversible
— used for serious violations or when a user permanently leaves the
institution. This distinction prevents accidental permanent cancellation
of identities that only need temporary restriction.

Q: What is TOTP and how did you implement it?
A: TOTP stands for Time-based One-Time Password. It uses a shared secret
between the server and an authenticator app (like Google Authenticator)
to generate a synchronized 6-digit code that changes every 30 seconds.
I implemented it using the speakeasy library. During setup, a secret is
generated, converted to a QR code for the user to scan into their app,
then a test code is verified before the secret is stored. On each login,
the submitted code is verified against the current and adjacent time
windows to account for clock drift.

Q: How does the audit trail support security in this system?
A: Every significant event — successful logins, failed attempts, OTP
activity, QR scan outcomes, identity lifecycle changes, admin actions —
is written to the AccessLog collection with the actor's ID, IP address,
browser information, and timestamp. This creates an immutable record
that allows administrators to investigate suspicious activity (e.g.
multiple failed scans of the same identity from different IPs) and
demonstrate accountability to institutional auditors.
